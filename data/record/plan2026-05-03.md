# AI 对话功能实现计划

## Context

Lightbulb AI 目前有 5 个功能模块（灵感提示、角色生图、三视图、海报、CG），缺少 AI 对话功能。用户需要在"灵感提示"旁新增一个"AI 对话"模块，支持流式对话、长期记忆（ChromaDB 向量检索 + 对话摘要），并复用现有模型管理体系通过 OpenAI SDK 直连大模型。

---

## 技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 大模型调用 | OpenAI SDK 直连 | 复用现有 ModelConfig + APIConfig 体系 |
| 向量数据库 | ChromaDB 本地服务 + `chromadb` npm 客户端 | 轻量、官方 JS 客户端、语义检索强 |
| 记忆策略 | 对话摘要 + 向量检索 | 长对话自动摘要，关键事实存向量库，新对话检索注入 |
| 流式传输 | SSE（Server-Sent Events） | 简单、HTTP 兼容、Vite 代理友好 |
| Embedding 模型 | `text-embedding-3-small`（通过当前 config 的 OpenAI 兼容端点） | 成本低、效果好 |
| Markdown 渲染 | `react-markdown` + `remark-gfm` | 轻量、GFM 支持 |

## 数据库变更

### 新增表（`backend/src/database.ts` → `initDatabase()`）

```sql
-- 对话表
CREATE TABLE IF NOT EXISTS conversations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT NOT NULL DEFAULT '新对话',
  model_provider TEXT NOT NULL,
  model_name TEXT NOT NULL,
  system_prompt TEXT,
  summary TEXT,
  summary_updated_at DATETIME,
  message_count INTEGER DEFAULT 0,
  is_archived INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 对话消息表
CREATE TABLE IF NOT EXISTS chat_messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  conversation_id INTEGER NOT NULL,
  role TEXT NOT NULL CHECK(role IN ('system','user','assistant')),
  content TEXT NOT NULL,
  token_usage INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_chat_messages_conv ON chat_messages(conversation_id);
```

- `getMessagesByConversationId(convId)` / `createMessage(data)` / `getMessageCount(convId)`

## 后端新文件

### 1. `backend/src/services/chatService.ts`
- `chatCompletion()` — 非流式，用于内部摘要/事实提取
- `chatCompletionStream()` — 流式，返回 AsyncIterable + getUsage()
- 复用 `createOpenAIClient(config)` 模式

### 2. `backend/src/services/embeddingService.ts`
- `generateEmbedding(text, config)` → `number[]`
- `generateEmbeddings(texts, config)` → `number[][]`
- `getEmbeddingConfig()` → 从 model_configs 中找支持文本的 config

### 3. `backend/src/services/chromaService.ts`
- `initChroma()` — 连接 `http://localhost:8000`，失败则降级（记忆功能关闭）
- `storeMemory({ conversationId, content, type })` — 存储摘要/事实
- `retrieveMemories({ query, nResults, conversationId })` — 检索相关记忆
- `deleteConversationMemories(conversationId)` — 删除对话关联记忆
- `isChromaAvailable()` — 检查可用性

### 4. `backend/src/services/memoryService.ts`
- `maybeSummarizeConversation(convId, config)` — 消息数 > 阈值时自动摘要
- `getRelevantMemories(userMessage, config, convId?)` → 注入 system prompt 的记忆文本
- `extractAndStoreFacts(convId, config)` — 提取关键事实存向量库

### 5. `backend/src/routes/chat.ts`

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/chat/conversations` | 对话列表（分页） |
| POST | `/api/chat/conversations` | 创建对话 |
| GET | `/api/chat/conversations/:id` | 获取对话 + 消息 |
| PUT | `/api/chat/conversations/:id` | 更新对话（标题/system_prompt） |
| DELETE | `/api/chat/conversations/:id` | 删除对话 + 消息 + 记忆 |
| POST | `/api/chat/conversations/:id/messages` | 发送消息 → SSE 流式响应 |
| POST | `/api/chat/conversations/:id/summarize` | 手动触发摘要 |

SSE 事件格式：
```
event: message_start  → { conversationId }
event: delta          → { content: "..." }
event: usage          → { promptTokens, completionTokens, totalTokens }
event: message_end    → { messageId, tokenUsage }
event: error          → { error: "..." }
```

## 记忆系统架构

```
用户发送消息
    │
    ▼
[1] 对用户消息生成 embedding
    │
    ▼
[2] 查询 ChromaDB: top 5 相似记忆
    │
    ▼
[3] 构建 system prompt (注入检索到的记忆)
    │
    ▼
[4] 组装 messages 数组: [system_prompt, ...历史消息, 新消息]
    │
    ▼
[5] 流式调用 OpenAI chat.completions
    │
    ▼
[6] 持久化 user + assistant 消息到 SQLite
    │
    ▼
[7] 异步检查: message_count > 20 且摘要过时?
    │ 是
    ▼
[8a] LLM 生成对话摘要 → 存 SQLite + ChromaDB
[8b] LLM 提取关键事实 → 存 ChromaDB
```

- 摘要触发阈值：消息 > 20 且距上次摘要新增 > 10 条
- 摘要后仍保留最近 20 条消息在上下文中，更早的由摘要代替
- ChromaDB 不可用时：整个记忆链路跳过，对话正常工作

## 验证方案

1. 启动后端 → 检查 conversations/chat_messages 表自动创建
2. 不启动 ChromaDB → 对话功能正常，控制台显示降级警告，UI 显示"记忆：关闭"
3. 启动 ChromaDB（`chroma run --path ./data/chromadb --port 8000`）→ 对话时记忆检索正常
4. 创建对话 → 发送消息 → 验证 SSE 流式响应
5. 发送 > 20 条消息 → 检查 SQLite 中 summary 字段已生成
6. 新对话中提到之前对话的关键信息 → 验证记忆检索注入
7. 前端 Markdown 渲染：发送包含表格/列表/代码块的消息
8. 中断流式响应（点击停止）→ 验证 AbortController 生效
9. 运行 `npm run lint` + `npm run format` 确认代码规范
